# 文章结构规范

本文件定义章节编号、案例、数据、表格、结论的结构要求。

## 章节编号

- **主节用中文数字编号 + `#` 标题**（一级标题）：`一、` `二、` `三、` ...。例如 `# 一、史上最快的 SaaS 神话`
- **子节用阿拉伯数字编号 + `##` 标题**（二级标题），编号格式为 `主节号.子节号`。例如 `## 2.1 从 GPT 到 Claude 的套壳术`、`## 2.2 套壳的三种死法`
- 文章不写总标题（没有单独的 H1 放文章题目），开篇直接进正文；主节的 `#` 就是文档最高一级标题
- 历史注：2026-04-19 之前的文章用的是 `##` / `###`，老文章不改；新文章起用新规则

## 不用 `---` 分隔线

- 章节之间不用水平分隔符 `---`。`#` / `##` 标题本身已经是分节信号，再插分隔线是冗余。
- 唯一允许 `---` 出现的地方：文章最后的 YAML-style 附录分隔（如 `## 作者其它文章`、`## 本文参考文献`、`## 附录：原始草稿` 之间的过渡）。且每段 `---` 前后必须是空行，总数不超过 3 条。
- 原因：Zhihu / 微信公众号 render 出来是一条灰色实线，视觉割裂感强，打断阅读流。

## 每一节写充分

宁可多不可少。用户反复反馈"写得太简单了""太短了"，所以每一节都要有足够的细节、数据、案例支撑，不要写提纲式的骨架。

## 平行子节深度对齐

一节下面如果有 N 个并列子节（4.1 / 4.2 / 4.3 / 4.4 这种），每个子节的段落数、数据密度、展开深度要大致相当。不允许第一个写得满，后面越写越短成提纲。

常见错误模式：**4.1 写了 3 段、有案例有数据；4.2 写了 2 段；4.3 只剩 1 段；4.4 2 段**。这种递减曲线在用户眼里就是"作者写累了"，整节的完整性会被拉低。

如果某个子节确实没那么多内容可写，合并进相邻子节或整节重新规划，不要留个空壳小节撑数量。

## 场景统一（多场景文章必守）

多个并列场景（例如"淘宝客服 + 餐厅大堂经理"）可以在开篇引子里并列出现，作为"这套模式跨行业通用"的证据。但**凡是进入详细展开、带完整步骤和数据的 worked example，必须统一到其中一条场景线**，不要在不同章节来回切换。

原因：读者在详细 example 里需要建立起对一个具体世界的理解（人物、数据源、业务规则、成本数字）。场景切换会让这层理解从头建，前后章节对不上号，整篇读起来割裂。

判断标准：如果两个详细 example 分别用了不同场景，问自己"这篇文章的主要工程示范是哪一个？"选一个，把另一个压缩回引子里点一下即可。

📍 **背景**：Harness Engineering 文章初版把 Context 层示例放在餐厅（大堂经理接待张先生），Harness 层示例放在淘宝（退款 agent 8 层壳），用户反馈例子要统一，后来全部聚焦淘宝线。

## 假设读者完全不懂

不要默认读者了解行业背景。涉及专业概念（如 SaaS、API、ARR、B2B）时，用一句话类比解释清楚再继续。例如"代码编辑器，你可以理解成程序员的 Word"。

## 案例要求

- **案例必须有名有姓有数据**。不要写"某公司做到了很高的收入"，要写"Pieter Levels 一个人做了 7 个产品，年入约 300 万美元，利润率超过 90%"
- **案例要新**。优先用 2024 到 2026 年的案例，避免用 2020 年之前的老故事
- **人物故事结构**：涉及人物案例时，先讲故事（起点低 → 失败 → 反转 → 成功），再总结方法论，让读者先被勾住再学东西

## 数据与对比

- **对比要用表格**。当需要对比两个公司、两种方案、两个时代时，用 Markdown 表格呈现，比纯文字清晰得多
- **融资、收入等时间线数据用列表或表格**，不要埋在段落里。读者需要能快速扫描关键数字

## 理论联系现实（科普文章必守）

每介绍一个理论概念后，紧跟着问"实验怎么说？"或"这和我们的世界有什么关系？"。如果没有实验验证，也要明确写出来，不要回避。

不要写成"数学理论报菜名"。

## 结论要有分量

- 不要用三四行草草收尾
- 好的结论应该：回扣开头、用新的比喻或金句重新点亮核心论点、给读者一个能记住的句子
- **结尾是 AI 味重灾区**：写完后按 `writing-rules.md` 里"结尾 AI 化陷阱"那条单独扫一遍（三项递进 / "只是序章" / "让我们" / "那个画面" 等）
